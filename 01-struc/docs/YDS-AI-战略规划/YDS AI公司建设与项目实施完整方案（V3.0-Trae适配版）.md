## 一、方案概述

### 1.1 项目背景与升级要点

随着AI技术在各行业的深度渗透，企业对智能化解决方案的需求呈爆发式增长。为抓住市场机遇，拟组建YDS AI公司，聚焦AI技术研发与落地服务。

**V3.0核心升级**：
- 基于Trae MCP协议构建智能体"虚拟会议室"系统
- 实现"异步协作 + 关键节点同步"的高效决策机制
- 建立类人化会议讨论能力，支持意见表达、分歧处理和共识达成
- 保持与现实公司岗位的无缝对接，确保AI公司可与传统企业协作

### 1.2 核心目标

1. **架构搭建**：3个月内完成基于Trae的智能体会议协作体系，实现各部门角色的数字化映射和会议决策能力。

2. **流程落地**：6个月内跑通"市场调研-方案设计-研发交付-运维收款"全业务流程，关键决策节点通过虚拟会议确保质量。

3. **业务验证**：12个月内完成3个以上标杆项目交付，验证AI会议决策的有效性和效率。

4. **效率提升**：通过智能体协作和会议优化，将项目交付周期缩短40%，决策质量提升30%，人力成本降低50%。

### 1.3 方案范围

本方案覆盖YDS AI公司从组织架构设计、Trae智能体会议体系搭建、异步协作机制配置到项目落地验证的全生命周期，重点包括：
- 分层融合架构设计（战略层、执行层、支撑层）
- 虚拟会议室系统构建
- 双重智能体角色体系（业务角色+技术角色）
- 端到端流程优化配置
- 决策追溯与学习机制

### 3.2 会议分级机制

**A级会议（战略决策）**：
- 参与者：战略层全体智能体 + 相关执行层代表
- 触发条件：重大项目决策、年度规划、危机处理
- 时长限制：60-90分钟
- 决策方式：充分讨论 + 投票表决
- 频率：月度或按需

**B级会议（业务协调）**：
- 参与者：执行层相关智能体
- 触发条件：项目启动、方案评审、问题解决
- 时长限制：30-45分钟
- 决策方式：专业讨论 + 共识达成
- 频率：周度或按需

**C级会议（日常沟通）**：
- 参与者：相关智能体小组
- 触发条件：任务协调、进度同步、技术讨论
- 时长限制：15-30分钟
- 决策方式：快速讨论 + 行动确认
- 频率：日度或按需

### 3.3 智能议程系统

**议程自动生成**：
1. 会议秘书智能体根据触发事件自动生成议程
2. 相关智能体可在会前24小时内补充议题
3. 系统自动排序议题优先级和预估讨论时间

**议程示例（项目启动会议）**：
```
会议主题：AI去水印工具项目启动会议
会议等级：B级会议
预计时长：45分钟
主持人：总经理智能体

议程：
1. 项目背景介绍 (企划部智能体, 5分钟)
2. 技术可行性分析 (技术部智能体, 10分钟)
3. 成本预算评估 (财务总监智能体, 8分钟)
4. 风险识别与应对 (全体讨论, 15分钟)
5. 项目计划确认 (项目协调者, 5分钟)
6. 行动项分配 (总经理智能体, 2分钟)
```

### 3.4 意见表达与分歧处理机制

**发言机制**：
- 轮流发言：按议程顺序，每个智能体有固定发言时间
- 举手发言：智能体可"举手"申请额外发言机会
- 紧急插话：遇到重大分歧时可申请紧急发言

**意见分类**：
- 🟢 **认同**：完全同意前述观点
- 🟡 **部分认同**：同意主要观点但有补充意见
- 🔴 **不同意**：明确反对并提出替代方案
- 🔵 **需要更多信息**：要求提供更多数据或分析

**分歧处理流程**：
1. **分歧识别**：系统自动检测意见冲突
2. **深度讨论**：给予分歧双方各3分钟阐述理由
3. **数据支撑**：要求提供客观数据或案例支持
4. **第三方意见**：邀请相关专业智能体提供建议
5. **投票决策**：无法达成共识时进行投票
6. **决策记录**：记录分歧点、讨论过程和最终决策依据

## 五、异步协作 + 关键节点同步机制

### 5.1 异步协作模式

**日常工作模式**：
- 各智能体按照既定流程异步执行任务
- 通过MCP协议实时同步工作状态和成果
- 共享工作区自动更新，保持信息透明
- 遇到问题时通过消息系统快速沟通

**异步协作优势**：
- 提高整体工作效率，避免等待时间
- 减少不必要的会议，专注核心决策
- 支持不同时区和工作节奏
- 降低协调成本，提升专业度

### 5.2 关键节点同步

**同步触发条件**：
1. **重大决策节点**：项目启动、方案变更、预算调整
2. **风险预警节点**：技术难题、进度延期、质量问题
3. **里程碑节点**：阶段完成、交付验收、项目复盘
4. **冲突解决节点**：意见分歧、资源冲突、优先级争议

**同步会议流程**：
```
会议前 (24小时)
├── 会议秘书智能体发送会议通知
├── 相关智能体准备发言材料
└── 系统自动收集相关数据和文档

会议中 (30-90分钟)
├── 主持人开场和议程确认 (2分钟)
├── 问题陈述和背景介绍 (5-10分钟)
├── 各方观点表达和讨论 (15-60分钟)
├── 分歧处理和共识达成 (5-15分钟)
└── 决策确认和行动项分配 (3-5分钟)

会议后 (2小时内)
├── 会议记录自动生成和分发
├── 行动项录入任务管理系统
└── 决策结果同步到相关智能体
```

## 六、端到端流程优化配置

### 6.1 需求阶段优化

**传统流程问题**：
- 需求理解不准确，反复修改
- 市场调研不充分，方案偏离实际
- 客户沟通不顺畅，期望管理不当

**优化后流程**：
1. **需求接收会议**（C级，15分钟）
   - 客户跟进智能体汇报客户需求
   - 产品经理智能体提出澄清问题
   - 确定需求调研计划

2. **异步需求分析**（1-2天）
   - 市场调研智能体进行行业分析
   - 竞品分析智能体研究同类产品
   - 产品经理智能体整理需求文档

3. **需求评审会议**（B级，30分钟）
   - 需求可行性讨论
   - 技术实现难度评估
   - 商业价值和成本分析
   - 确定项目是否启动

### 6.2 设计阶段优化

**设计协作会议**（B级，45分钟）：
- 产品经理智能体展示原型设计
- UI设计智能体提供视觉方案
- 前端开发智能体评估实现难度
- 用户体验专家提出优化建议

**异步设计完善**：
- 基于会议反馈优化设计方案
- 技术可行性验证和调整
- 设计规范和组件库建立

### 6.3 开发阶段优化

**开发启动会议**（B级，30分钟）：
- 项目协调者制定开发计划
- 前后端智能体确认接口设计
- 测试智能体制定测试策略

## 十二、Trae适配技术实施细则（新增）

### 12.1 MCP接口层MVP统一消息模型

为支撑“语音/文字/文档共享/投票”多通道会议协作，定义统一消息模型（关键字段一致，按通道携带不同payload）：

```json
{
  "id": "msg-2025-11-03-0001",
  "room_id": "room-2025-001",
  "channel": "text | voice | docs | vote",
  "event_type": "message.text | voice.stream | docs.share | vote.cast",
  "topic": "rooms/{room_id}/{channel}/{event_type}",
  "sender": { "id": "agent.ceo", "role": "CEO", "display_name": "总经理智能体" },
  "timestamp": "2025-11-03T09:30:10Z",
  "payload": {
    "text": { "content": "讨论要点……" },
    "voice": { "session_id": "voice-abc", "chunk_seq": 1, "sample_rate": 16000, "format": "pcm16", "data_base64": "..." },
    "docs": { "action": "share", "path": "S:\\YDS-Lab\\Struc\\SharedWorkspace\\Documents\\MeetingAttachments\\demo.pdf", "name": "demo.pdf", "type": "pdf", "size": 12345, "hash": "sha256:...", "permissions": "read" },
    "vote": { "proposal_id": "vote-001", "options": ["A", "B"], "selected": "A", "anonymous": false, "weight": 1, "quorum": 0.6, "expires_at": "2025-11-03T10:00:00Z" }
  }
}
```

说明：payload为“按需”的并集结构，随channel取其子对象；客户端和服务端需校验event_type与payload一致性。

### 12.2 端点清单与路由（规划）

- WebSocket：`/ws?token=...`
  - 事件：`message.text`、`voice.stream`、`docs.share`、`vote.cast`、`system.notice`
  - Topic：`rooms/{room_id}/{channel}/{event_type}`

- REST（MCP控制面，规划中的接口）：
  - `POST /mcp/messages/send`
  - `POST /mcp/docs/share`、`GET /mcp/docs/list`
  - `POST /mcp/votes/create`、`POST /mcp/votes/cast`、`GET /mcp/votes/status`
  - `POST /mcp/voice/session/start`、`POST /mcp/voice/chunk`

路由策略：以`room_id`为主键、`channel`为维度进行发布-订阅；语音通道采用分片（chunk）顺序传输，服务端进行重组与STT处理。

### 12.3 与现有meetingroom_server.py的对接改造

- 建立统一事件管道：在`socketio.emit`层统一封装消息模型，确保前端/后端一致。
- 新增REST控制面：为文档共享、投票、语音会话提供`/mcp/*`端点（鉴权后使用）。
- 语音链路：`voice.stream`事件桥接至本机STT（Shimmy优先，云端为备选），支持弱网重传与分片校验。
- 前端目录回退：已支持回退到`Struc/MarketingDept/Task/003-meetingroom/ui`，后续在UI层补齐文档共享与投票组件。

### 12.4 权限与审计（MVP）

- 身份认证：采用JWT（或同等机制），在WS握手与REST调用均校验`token`。
- RBAC角色：CEO、会议秘书、财务总监、技术代表、观察员等；按角色授予通道级操作权限。
- 资源级ACL：对文档共享引入路径白名单与黑名单；对“合同/对外原件”默认禁止共享，仅允许摘要引用。
- 审计留痕：记录每次共享与投票的操作者、时间、资源路径与校验结果，输出至`01-struc/0B-general-manager/logs/audit_trails`与会议留存。

### 12.5 数据留存与压缩

- 会议记录：文本与关键事件（含投票结果、共享文档摘要）进入会议纪要；语音原始分片不默认留存，仅保留转写文本。
- 压缩策略：日志与纪要按月归档，必要时进行zip压缩与哈希校验；敏感摘要加水印。

## 十三、文档共享治理与目录白名单（新增）

### 13.1 分类与用途边界

- 机密类（只读引用，禁止原件共享）：合同原件、对外正式文件原稿等。
- 共享类（会议附件与协作文档主目录）：会议附件、内部说明、任务材料。

### 13.2 允许目录清单（白名单）

为避免`01-struc/docs`下的敏感资料误共享，会议系统默认不允许该目录作为共享源，除明确单文件只读引用外。推荐白名单如下：

1. `S:\YDS-Lab\01-struc\SharedWorkspace\Documents\MeetingAttachments\**`（读/写/共享）
2. `S:\YDS-Lab\01-struc\SharedWorkspace\Collaboration\**`（读/写/协作）
3. `S:\YDS-Lab\output\011-showcase-materials\assets\**`（只读共享）
4. `S:\YDS-Lab\01-struc\docs\YDS-AI-战略规划\*.md`（只读引用，禁止导出原稿）
5. 默认禁止：`S:\YDS-Lab\01-struc\docs\**\合同\**`与名称包含`合同/对外文档`的路径（可由财务总监提交“去敏摘要”到共享目录后再引用）。

### 13.3 RBAC与path_restrictions权限矩阵（示例）

- CEO：可读取“共享类/对外展示类”，机密类仅摘要只读；可发起投票；可审批共享。
- 会议秘书：可读写附件目录、生成会议纪要；无权访问合同原件路径；可撤回误共享。
- 财务总监：可读取合同摘要材料（由行政/秘书去敏后置于共享目录）；不可直接共享原件。
- 技术代表/观察员：只读共享类与对外展示类；无写权限；禁止访问机密类。

## 十四、会议语音服务（Shimmy/Ollama）部署与配置（新增）

### 14.1 环境与路径

- 建议在本机设置：`SHIMMY_HOME = S:\\LLM\\models\\shimmy`（如存在历史兜底`S:\\LLm`路径，统一迁移至`S:\\LLM`）。
- 将`shimmy.exe`加入PATH；确保`meetingroom_server.py`能自动发现Shimmy服务。

### 14.2 external_models.json 映射（示例）

```json
{
  "stt": {
    "default": "shimmy.whisper-tiny-zh",
    "providers": {
      "shimmy": { "host": "http://127.0.0.1:51080", "models": ["whisper-tiny-zh", "whisper-small-zh"] },
      "azure": { "endpoint": "https://<your-azure-stt>", "key": "<secure>" }
    }
  },
  "tts": {
    "default": "shimmy.onnx-zh",
    "providers": {
      "shimmy": { "host": "http://127.0.0.1:51080", "voices": ["zh_female", "zh_male"] }
    }
  },
  "llm": {
    "default": "ollama.qwen2.5",
    "providers": {
      "ollama": { "host": "http://127.0.0.1:11434", "models": ["qwen2.5:latest", "llama3.1:8b"] }
    }
  }
}
```

### 14.3 端口与健康检查

- Ollama默认端口：`11434`；Shimmy本地服务建议端口：`51080`。
- 健康检查：
  - `Invoke-WebRequest http://127.0.0.1:11434/api/tags`（Ollama模型可用性）
  - `Invoke-WebRequest http://127.0.0.1:51080/health`（Shimmy健康状态）

### 14.4 语音链路参数（建议）

- 分片大小：32–64KB；采样率：16k；格式：PCM16（或WebM/Opus经服务端转换）。
- 弱网模式：启用重传与分片校验；语音转写超时与断点续传策略。
- 云优先策略：本地Shimmy为备份；云STT失败或不可用时自动回退至本地。

### 14.5 MVP闭环测试

- 客户端采集→WS推送`voice.stream`→服务端重组→STT→转写文本回传→会议纪要生成。
- 记录端到端时延与准确率，输出到`01-struc/0B-general-manager/logs/performance_metrics`。
