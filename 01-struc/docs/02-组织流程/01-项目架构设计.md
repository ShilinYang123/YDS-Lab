> **适用范围**: YDS-Lab全系统

## 架构概述

YDS AI公司采用AI原生的分布式协作架构设计，通过Trae IDE集成多智能体系统（20个专业AI Agent），结合标准化的目录结构和自动化工具链，实现"一个人+一群AI"的低门槛创业模式。系统以AI Agent协作和项目生命周期管理为核心，提供从任务规划到产品交付的全流程智能化解决方案。

最新架构采用v2.3版本，核心特色是**项目生命周期管理体系**：从任务规划（02-task/）→开发实现（03-dev/）→生产部署（04-prod/），实现全流程管控。同时通过01-struc/组织架构模拟真实公司运作，提供完整的AI协作环境。

### 🏢 三级存储规范（v2.3核心规范）
系统采用严格的三级存储架构，确保文件管理规范化：

**📁 bak - 一级备份存储**
- 全公司只建立一级备份目录：`S:\YDS-Lab\bak`
- 所有备份必须集中存储，不允许零散备份
- 禁止备份膨胀，统一备份策略和管理

**📁 docs - 三级文档存储**
- **公司级文档**：`01-struc/docs/` - 存放公司级战略规划、组织流程等核心文档
- **项目级文档**：`项目名/docs/` - 各项目的专用文档存储  
- **系统级文档**：`docs/` - 系统维护和其他通用文档

**📁 logs - 三级日志存储**  
- **公司级日志**：`01-struc/logs/` - 公司级操作日志、系统日志
- **项目级日志**：`项目名/logs/` - 各项目的运行日志
- **系统级日志**：`logs/` - 系统维护和通用日志

**📁 rep - 三级报告存储**
- **公司级报告**：`01-struc/rep/` - 公司级质量报告、审计报告等
- **项目级报告**：`项目名/rep/` - 各项目的专项报告
- **系统级报告**：`rep/` - 系统维护和通用报告

### 2.2 三级存储规范（v2.3严格执行）

#### 2.2.1 备份存储（bak）
- **全公司只建立一级**：`\bak`
- **所有备份必须存于此**，不允许零散备份、膨胀备份
- **统一管理**：所有备份操作必须经过审批并记录

#### 2.2.2 文档存储（docs）三级架构
- **公司级文档**：`S:\YDS-Lab\01-struc\docs`（公司战略、组织架构、规范制度）
- **项目级文档**：`项目名\docs`（具体项目的设计文档、需求文档）
- **系统维护文档**：`S:\YDS-Lab\docs`（系统维护、操作手册、技术文档）

#### 2.2.3 日志存储（logs）三级架构（规则同docs）
- **公司级日志**：`S:\YDS-Lab\01-struc\logs`（公司运营日志、审计日志）
- **项目级日志**：`项目名\logs`（项目运行日志、调试日志）
- **系统维护日志**：`S:\YDS-Lab\logs`（系统维护、运维日志、临时工具运行日志）

**重要补充说明**：
- **系统维护类工具产物**：所有系统维护、编码检查、环境验证等工具的运行结果和日志，统一存储在`S:\YDS-Lab\logs`目录
- **临时性工具产物**：各种脚本工具、检查工具、验证工具产生的临时报告和日志，归类为系统维护日志
- **工具脚本分类**：系统维护工具（如enco.py、check_structure.py等）不属于公司级战略文档，其运行产物应存储在系统维护日志目录

#### 2.2.4 报告存储（rep）三级架构（规则同docs）
- **公司级报告**：`S:\YDS-Lab\01-struc\rep`（公司级质量报告、合规报告、审计报告）
- **项目级报告**：`项目名\rep`（项目进度报告、测试报告、验收报告）
- **系统维护报告**：`S:\YDS-Lab\rep`（系统性能报告、故障分析报告、临时工具报告）

**重要补充说明**：
- **系统维护类工具报告**：所有系统维护、编码检查、环境验证等工具的运行报告，统一存储在`S:\YDS-Lab\rep`目录
- **临时性工具报告**：各种脚本工具、检查工具、验证工具产生的临时报告，归类为系统维护报告
- **工具脚本分类原则**：
  - **公司级**：公司战略、组织架构、重大决策类报告 → `01-struc/rep/`
  - **项目级**：具体项目的进度、测试、验收报告 → `项目名/rep/`
  - **系统维护级**：工具运行结果、系统检查报告、临时验证报告 → `logs/` 或 `rep/`

### 2.3 工具脚本产物存储规范（v2.3新增）

#### 2.3.1 分类原则
- **部门只放部门必须的内容**：各部门目录只存放该部门核心职能相关的文件
- **公司里放公司战略及决策宏观内容**：公司级目录只存放战略、规范、制度等宏观管理文件
- **项目里只放本项目密切需要的内容**：项目级目录只存放该项目直接相关的开发、测试、部署文件
- **其余都归为系统维护和其他临时类**：其他所有工具运行产物、临时文件、系统检查报告等统一归类到系统维护级别

#### 2.3.2 工具脚本产物存储规则
| 工具类型 | 产物类型 | 存储位置 | 说明 |
|---------|---------|---------|------|
| 编码检查工具（如enco.py） | 检查报告、日志 | `S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/` | 系统维护类，不属于公司战略级 |
| 结构检查工具 | 检查报告 | `S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/` | 系统维护类，临时性工具产物 |
| 环境验证工具 | 验证报告 | `S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/` | 系统维护类，临时性工具产物 |
| 系统监控工具 | 监控日志 | `S:\YDS-Lab\logs/` | 系统维护类，运行日志 |
| 备份工具 | 备份日志 | `S:\YDS-Lab\logs/` | 系统维护类，操作日志 |
| 部署工具 | 部署报告 | `S:\YDS-Lab\rep/` | 系统维护类，部署结果报告 |

#### 2.3.3 存储位置判定标准
- **是否属于公司战略决策？** → 是：`01-struc/rep/`，否：继续判断
- **是否属于具体项目交付？** → 是：`项目名/rep/`，否：继续判断  
- **是否属于系统维护/临时工具？** → 是：`S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/`，否：重新评估分类

#### 1.2 Trae Agent协作机制

基于Trae IDE的多智能体协作系统，采用轻量级编排器实现Agent间协作：

**核心组件**:
- **协作编排器**: `tools/agents/run_collab.py` - 轻量级多Agent协作编排
- **LLM路由**: `models/services/llm_router.py` - 统一模型调用接口
- **Agent配置**: `Struc/Agents/*/prompt.py` - 各部门Agent专用提示词
- **会议归档**: `Struc/Agents/ceo/tools.py::archive_meeting` - 自动归档机制

**协作模式**:
- **任务自动分发**: 总经理Agent根据项目需求自动分配任务给相应专业Agent
- **跨部门协作**: Agent间通过Trae Agent平台进行信息共享和协作
- **智能决策支持**: 基于多Agent分析结果提供决策建议
- **自适应学习**: 根据项目反馈优化Agent行为和协作模式
- **会议协作**: 支持每日晨会和紧急会议，自动生成标准化会议纪要
- **轻量编排**: 使用tools/agents/run_collab.py实现不依赖CrewAI的协作编排

**技术特色**:
- **不依赖CrewAI**: 采用自主开发的轻量协作编排器
- **标准化输出**: 自动生成Markdown和JSON格式的会议纪要
- **行动项追踪**: 专用表格格式记录决策事项和责任人
- **定时执行**: 支持Windows任务计划程序定时执行

##### （1）按专长分配
- 技术任务分配给技术部智能体
- 财务任务分配给财务部智能体
- 市场任务分配给市场部智能体

##### （2）按优先级分配
- 高优先级任务优先分配给高级智能体
- 紧急任务分配给响应速度快的智能体
- 复杂任务分配给协作能力强的智能体组

##### （3）按负载均衡
- 避免单个智能体任务过载
- 根据智能体当前负载动态调整
- 保持整体协作效率最优化
