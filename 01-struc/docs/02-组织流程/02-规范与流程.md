## 项目概述

YDS-Lab是一个基于AI原生分布式协作架构的智能开发平台，通过Trae IDE集成多智能体系统，结合标准化的目录结构和自动化工具链，实现项目从开发到交付的全生命周期管理。系统以AI协作和项目管理为核心，提供高效、智能的开发解决方案。

### 项目存储位置（YDS-Lab标准架构V5.1）
- **任务规划位置**：S:\YDS-Lab\02-task（任务规划与工作区）
- **开发工作位置**：S:\YDS-Lab\03-dev（项目开发工作区）
- **生产部署位置**：S:\YDS-Lab\04-prod（生产环境部署区）
- **组织架构位置**：S:\YDS-Lab\01-struc（公司组织架构与AI协作环境）
- **客户交付位置**：S:\YDS-Lab\04-prod\deliverables（客户项目交付成果区）
- **通用工具位置**：S:\YDS-Lab\tools（通用工具库）

### 技术愿景
- **AI原生架构**：以Trae IDE为核心平台，实现多智能体协同工作
- **分布式协作**：通过01-struc/组织架构模拟真实公司运作，提供完整的AI协作环境
- **项目生命周期管理**：从任务规划（02-task/）→开发实现（03-dev/）→生产部署（04-prod/）的全流程管控
- **标准化开发**：统一的目录结构、命名规范和工作流程
- **智能化运维**：简化工具链（ch、fi、st、up）支持环境检查、备份恢复和系统维护

### 架构原则
- **分层设计**：采用AI智能体层、项目管理层、工具支撑层的分层架构
- **模块化**：各部门工作区按功能分类管理，支持独立开发和扩展
- **标准化**：统一的目录结构和数据交换格式
- **可扩展**：支持新增项目类型和功能模块
- **高可用**：完善的错误处理和故障恢复机制

## 一、目录结构管理规范

### 1. 标准目录结构（基于三级存储规范）

YDS-Lab项目采用标准化的目录结构，**严格执行三级存储规范**：

```
S:\YDS-Lab\
├── 01-struc\                # 公司组织架构与AI协作环境（公司级存储）
│   ├── Agents\              # AI智能体配置与管理
│   ├── docs\                # 公司级文档存储（战略、规范、制度）
│   ├── logs\                # 公司级日志存储（运营、审计日志）
│   ├── rep\                 # 公司级报告存储（质量、合规、审计报告）
│   └── bak\                 # 全公司统一备份存储（严禁零散备份）
├── 02-task\                # 任务规划与工作区
├── 03-dev\                 # 项目开发工作区
│   ├── 项目名\              # 具体项目目录
│   │   ├── docs\            # 项目级文档存储
│   │   ├── logs\            # 项目级日志存储
│   │   ├── rep\             # 项目级报告存储
│   │   └── src\             # 项目源代码
│   └── templates\          # 项目模板
├── 04-prod\                # 生产环境部署区
│   ├── 项目名\              # 生产项目目录
│   │   ├── docs\            # 生产项目文档
│   │   ├── logs\            # 生产项目日志
│   │   └── rep\             # 生产项目报告
│   └── deliverables\       # 客户交付成果
├── docs\                    # 系统维护文档存储
│   ├── docs\                # 系统文档
│   ├── logs\                # 系统维护日志
│   └── rep\                 # 系统维护报告
├── tools\                   # 自动化工具集
│   ├── check\              # 检查工具
│   ├── config\             # 配置管理
│   ├── mcp\                # MCP服务器工具
│   └── scripts\            # 脚本工具
├── models\                  # AI模型配置与路径映射
└── bak\                    # 系统级备份存储（统一备份策略）
```

**重要说明**：
- **bak**：全公司只建立一级，所有备份必须存于此，不允许零散备份
- **docs**：三级架构 - 公司级(01-struc/docs)、项目级(项目名/docs)、系统级(docs/)
- **logs**：三级架构 - 公司级(01-struc/logs)、项目级(项目名/logs)、系统级(logs/)
- **rep**：三级架构 - 公司级(01-struc/rep)、项目级(项目名/rep)、系统级(rep/)

**目录扫描特殊处理规则**：
- **bak目录**：结构检查工具不扫描内部文件，仅验证目录存在（max_depth: 0, show_files: False）
- **logs目录**：结构检查工具不扫描内部文件，仅验证目录存在（max_depth: 0, show_files: False）
- **rep目录**：结构检查工具不扫描内部文件，仅验证目录存在（max_depth: 0, show_files: False）
- **archives目录**：结构检查工具仅显示一级子目录，不显示具体文件（max_depth: 1, show_files: False）

**规则原理**：存储类目录（bak、logs、rep）的内部文件内容不影响项目结构合规性，结构检查工具只需验证这些目录的存在性。archives目录同理，只需验证一级子目录结构。这种处理方式既保证了目录结构的完整性检查，又避免了存储目录内部文件对结构检查结果的干扰。

### 2. 三级存储规范（严格执行）

#### 2.1 备份存储（bak）
- **全公司只建立一级**：`\bak`
- **所有备份必须存于此**，不允许零散备份、膨胀备份
- **统一管理**：所有备份操作必须经过审批并记录

#### 2.2 文档存储（docs）三级架构
- **公司级文档**：`S:\YDS-Lab\01-struc\docs`（公司战略、组织架构、规范制度）
- **项目级文档**：`项目名\docs`（具体项目的设计文档、需求文档）
- **系统维护文档**：`S:\YDS-Lab\docs`（系统维护、操作手册、技术文档）

#### 2.3 日志存储（logs）三级架构（规则同docs）
- **公司级日志**：`S:\YDS-Lab\01-struc\logs`（公司运营日志、审计日志）
- **项目级日志**：`项目名\logs`（项目运行日志、调试日志）
- **系统维护日志**：`S:\YDS-Lab\logs`（系统维护、运维日志、临时工具运行日志）

**重要补充说明**：
- **系统维护类工具产物**：所有系统维护、编码检查、环境验证等工具的运行结果和日志，统一存储在`S:\YDS-Lab\logs`目录
- **临时性工具产物**：各种脚本工具、检查工具、验证工具产生的临时报告和日志，归类为系统维护日志
- **工具脚本分类**：系统维护工具（如enco.py、ch.py等）不属于公司级战略文档，其运行产物应存储在系统维护日志目录

#### 2.4 报告存储（rep）三级架构（规则同docs）
- **公司级报告**：`S:\YDS-Lab\01-struc\rep`（公司级质量报告、合规报告、审计报告）
- **项目级报告**：`项目名\rep`（项目进度报告、测试报告、验收报告）
- **系统维护报告**：`S:\YDS-Lab\rep`（系统性能报告、故障分析报告、临时工具报告）

**重要补充说明**：
- **系统维护类工具报告**：所有系统维护、编码检查、环境验证等工具的运行报告，统一存储在`S:\YDS-Lab\rep`目录
- **临时性工具报告**：各种脚本工具、检查工具、验证工具产生的临时报告，归类为系统维护报告
- **工具脚本分类原则**：
  - **公司级**：公司战略、组织架构、重大决策类报告 → `01-struc/rep/`
  - **项目级**：具体项目的进度、测试、验收报告 → `项目名/rep/`
  - **系统维护级**：工具运行结果、系统检查报告、临时验证报告 → `logs/` 或 `rep/`

### 2.5 工具脚本产物存储规范（新增）

#### 2.5.1 分类原则
- **部门只放部门必须的内容**：各部门目录只存放该部门核心职能相关的文件
- **公司里放公司战略及决策宏观内容**：公司级目录只存放战略、规范、制度等宏观管理文件
- **项目里只放本项目密切需要的内容**：项目级目录只存放该项目直接相关的开发、测试、部署文件
- **其余都归为系统维护和其他临时类**：其他所有工具运行产物、临时文件、系统检查报告等统一归类到系统维护级别

#### 2.5.2 工具脚本产物存储规则
| 工具类型 | 产物类型 | 存储位置 | 说明 |
|---------|---------|---------|------|
| 编码检查工具（如enco.py） | 检查报告、日志 | `S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/` | 系统维护类，不属于公司战略级 |
| 结构检查工具 | 检查报告 | `S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/` | 系统维护类，临时性工具产物 |
| 环境验证工具 | 验证报告 | `S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/` | 系统维护类，临时性工具产物 |
| 系统监控工具 | 监控日志 | `S:\YDS-Lab\logs/` | 系统维护类，运行日志 |
| 备份工具 | 备份日志 | `S:\YDS-Lab\logs/` | 系统维护类，操作日志 |
| 部署工具 | 部署报告 | `S:\YDS-Lab\rep/` | 系统维护类，部署结果报告 |

#### 2.5.3 存储位置判定标准
- **是否属于公司战略决策？** → 是：`01-struc/rep/`，否：继续判断
- **是否属于具体项目交付？** → 是：`项目名/rep/`，否：继续判断  
- **是否属于系统维护/临时工具？** → 是：`S:\YDS-Lab\logs/` 或 `S:\YDS-Lab\rep/`，否：重新评估分类

### 3. 目录命名规范

- **项目目录**：使用英文名称，采用PascalCase或kebab-case
- **文档目录**：中英文结合，便于理解和维护
- **工具脚本**：使用snake_case命名，功能明确
- **配置文件**：统一使用.json或.yaml格式
- **存储目录**：严格按照三级规范执行，不得随意创建同级目录

### 4. 文件权限管理

- **核心工具脚本**：ch.py、up.py等设为只读
- **配置文件**：由tools/control.py统一管理权限
- **项目文件**：开发人员具有完整读写权限
- **系统日志**：只读权限，由系统自动维护

## 四、AI协作规范

### 1. 智能体配置规范

#### （1）AI Agent分类
- **战略决策智能体**：CEO、CFO、战略顾问等
- **业务执行智能体**：企划、产品、技术、运营等
- **基础支撑智能体**：行政、知识库、安全合规等

#### （2）配置管理
- 统一配置文件格式：YAML格式
- 配置存放位置：`01-struc/Agents/`目录下
- 命名规范：`{序号}-{角色名称}/`

#### （3）协作模式
- **虚拟会议室**：基于Trae MCP协议构建
- **异步协作**：日常任务异步处理
- **关键节点同步**：重要决策点同步讨论

### 2. 任务分配原则

#### （1）按专长分配
- 技术任务分配给技术部智能体
- 财务任务分配给财务部智能体
- 市场任务分配给市场部智能体

#### （2）按优先级分配
- 高优先级任务优先分配给高级智能体
- 紧急任务分配给响应速度快的智能体
- 复杂任务分配给协作能力强的智能体组

#### （3）按负载均衡
- 避免单个智能体任务过载
- 根据智能体当前负载动态调整
- 保持整体协作效率最优化

## 五、代码开发规范

### 1. 编程规范

#### （1）Python代码规范
- 遵循Python PEP 8编码规范
- 使用类型注解提高代码可读性
- 函数和类必须有文档字符串
- 复杂逻辑必须添加注释说明

#### （2）文件组织规范
- 模块文件按功能分类存放
- 避免单个文件过大（超过500行需要拆分）
- 统一使用UTF-8无BOM编码格式
- 文件头部添加编码声明：`# -*- coding: utf-8 -*-`

#### （3）命名规范
- 变量名使用snake_case
- 类名使用PascalCase
- 常量名使用UPPER_CASE
- 私有成员以单下划线开头

### 2. 版本控制规范

#### （1）Git使用规范
- 提交信息必须清晰描述修改内容
- 类型包括：feat、fix、docs、style、refactor、test等
- 每次提交只包含一个逻辑变更
- 禁止提交敏感信息和密钥

#### （2）分支管理规范
- main分支：稳定版本，禁止直接提交
- develop分支：开发版本，集成测试通过
- feature分支：功能开发，完成后合并到develop
- hotfix分支：紧急修复，完成后合并到main和develop

#### （3）Git忽略文件规范
- 项目使用统一的.gitignore文件（位于项目根目录）
- 禁止提交以下类型的文件：
  - 备份文件：bak/目录、*.bak、*.backup等
  - 报告文件：工作记录/目录、*.report等
  - 记录文件：rep/目录、*.record等
  - 临时文件：tmp/目录、*.tmp、cache/目录等
  - 日志文件：logs/目录、*.log等
  - 虚拟环境：venv/目录、.venv/目录等
  - IDE配置：.vscode/目录、.idea/目录等
  - 监控文件：monitoring/目录及其子目录
  - 培训文件：training/目录及其子目录
  - 大文件：*.zip、*.tar.gz、*.exe等（超过10MB）
  - 敏感信息：*.key、*.pem、*.env等
- 详细规则请参考项目根目录下的.gitignore文件
- 所有Git操作必须遵循此忽略规则，确保不提交敏感或不必要的文件

### 3. 质量保证规范

#### （1）代码审查
- 所有代码必须经过同行审查
- 审查重点：逻辑正确性、性能优化、安全漏洞
- 审查记录必须完整保存
- 处理检查报告中的违规项

#### （2）测试规范
- 单元测试覆盖率不低于80%

**工具定位**：YDS-Lab目录结构合规性检查核心工具，V5.2版本与up.py实现完全统一，确保检查结果一致性。

**检查维度**：
1. **合规性检查**：与《02-项目架构设计.md》标准规范对比，计算合规率
2. **白名单一致性**：与up.py生成的《动态目录结构清单》.md对比，识别具体差异项

**报告输出**：
- **合规率统计**：总标准项目数、实际项目数、合规项目数、缺失项目数、额外项目数、合规率百分比

- **交叉验证**：识别假缺失和假额外项目，确保检查结果准确性

**重要说明：目录扫描特殊处理规则**
- **bak目录**：不扫描内部文件，仅显示目录本身（max_depth: 0, show_files: False）
- **logs目录**：不扫描内部文件，仅显示目录本身（max_depth: 0, show_files: False）
- **rep目录**：不扫描内部文件，仅显示目录本身（max_depth: 0, show_files: False）
- **archives目录**：仅显示一级子目录，不显示具体文件（max_depth: 1, show_files: False）
- **04-prod目录**：显示二级子目录结构，允许临时文件存在（max_depth: 2, show_files: True, allow_temp_content: True）

**规则说明**：
- bak、logs、rep目录作为存储类目录，其内部文件内容不影响项目结构合规性，因此结构检查工具只验证这些目录的存在性，不检查其内部文件
- archives目录同理，只验证一级子目录结构
- 04-prod目录作为生产环境部署区，需要展示二级目录结构以便监控项目部署状态，同时允许存在运行时临时文件（如.env.production、临时日志等），这些文件不影响结构合规性检查
